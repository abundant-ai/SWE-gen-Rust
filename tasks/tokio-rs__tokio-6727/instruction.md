When using Tokio with tokio-console/console-subscriber, completed tasks can incorrectly appear as still “alive” and show a warning that the task has “lost its waker”. This happens because some Tokio runtime resources create tracing spans that implicitly inherit the currently active span as their parent (the tracing contextual parent). If a resource (like a timer or semaphore) is created inside a task and then outlives that task (for example, because it’s moved into another task), the resource span keeps the original task’s span alive longer than it should be. As a result, tools that interpret the tracing graph (such as tokio-console) think the originating task is still present, even though it has completed.

This is reproducible by spawning a task that awaits a short sleep and then spawns another task that sleeps for a long time. The first task should complete quickly, but tokio-console may display it as having lost its waker instead of being cleanly completed.

The tracing instrumentation for Tokio resources should not artificially extend a task span’s lifetime. Resource spans for `tokio::time::Sleep` and for the synchronization primitive `tokio::sync::batch_semaphore::BatchSemaphore` must be created as explicit root spans (i.e., with no parent), rather than inheriting the currently active contextual parent span.

Concretely, creating a `tokio::time::sleep(Duration::from_millis(...))` should emit a `runtime.resource` span targeted at `tokio::time::sleep` that has an explicit parent of `None`. Similarly, creating structures that internally construct a `BatchSemaphore` (for example `tokio::sync::Mutex::new(...)`) should result in a `runtime.resource` span targeted at `tokio::sync::batch_semaphore` that also has an explicit parent of `None`.

This change should ensure that tasks which create these resources can finish without their spans being kept alive by resource spans, eliminating the spurious “lost its waker” warning in tokio-console for tasks that have actually completed.
